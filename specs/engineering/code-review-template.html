功能实现审查 ​

[ ] 功能正确性 ​

  • [ ] 功能是否完全符合需求文档要求
  • [ ] 是否覆盖了所有边界条件和异常情况
  • [ ] 是否有适当的输入验证
  • [ ] 输出结果是否符合预期

[ ] 业务逻辑 ​

  • [ ] 业务逻辑是否清晰易懂
  • [ ] 是否符合业务规则和流程
  • [ ] 是否考虑了并发和竞态条件
  • [ ] 是否有适当的事务处理

代码质量审查 ​

[ ] 编码规范 ​

  • [ ] 是否遵循团队编码规范
  • [ ] 命名是否清晰有意义
  • [ ] 代码格式是否一致
  • [ ] 是否有适当的注释

[ ] 设计质量 ​

  • [ ] 是否遵循SOLID原则
  • [ ] 函数和类的职责是否单一
  • [ ] 是否有适当的抽象和封装
  • [ ] 是否合理使用设计模式

[ ] 复杂度控制 ​

  • [ ] 函数长度是否合理（建议不超过50行）
  • [ ] 圈复杂度是否过高（建议不超过10）
  • [ ] 嵌套层级是否过深（建议不超过3层）
  • [ ] 是否有重复代码

性能审查 ​

[ ] 性能考量 ​

  • [ ] 算法时间复杂度是否合理
  • [ ] 是否有不必要的计算或查询
  • [ ] 内存使用是否合理
  • [ ] 是否考虑了缓存机制

[ ] 资源管理 ​

  • [ ] 文件、网络连接等资源是否正确释放
  • [ ] 是否有内存泄漏风险
  • [ ] 是否合理使用异步处理
  • [ ] 数据库查询是否优化

安全性审查 ​

[ ] 安全考虑 ​

  • [ ] 是否有SQL注入风险
  • [ ] 是否有XSS攻击风险
  • [ ] 用户输入是否正确过滤和转义
  • [ ] 敏感信息是否妥善处理

可维护性审查 ​

[ ] 可维护性 ​

  • [ ] 代码是否易于理解和修改
  • [ ] 是否有适当的日志记录
  • [ ] 错误处理是否完善
  • [ ] 是否易于测试

[ ] 文档完整性 ​

  • [ ] 是否更新了相关文档
  • [ ] API文档是否完整
  • [ ] README是否更新
  • [ ] 注释是否清晰

测试审查 ​

[ ] 测试覆盖 ​

  • [ ] 单元测试是否覆盖核心逻辑
  • [ ] 是否测试了边界条件
  • [ ] 是否测试了异常情况
  • [ ] 测试用例是否清晰易懂

[ ] 测试质量 ​

  • [ ] 测试是否独立且可重复
  • [ ] 是否使用了适当的测试框架
  • [ ] 测试数据是否合理
  • [ ] 是否有性能测试
